35 Claude Code Commands, Tricks & Workflows Most Users Don't Know

AI · Claude Code · 35 techniques, five categories

The Big Idea

Most developers install Claude Code, use it for basic code generation, and assume they've seen what it can do — they've seen maybe 20%. The other 80% is what turns it from a useful assistant into the most productive tool in your terminal. Compiled by Khairallah AL-Awady from months of daily use. Tested daily. Zero theory.

💡 Key insight

The single highest-leverage habit on this list: plan before you build. Use plan mode and incremental, tested steps — it prevents more bugs than any other technique here.

Essential Commands (01–08)

#CommandWhat it does
01Plan Mode Shift + TabClaude analyzes your codebase and writes an architecture plan without writing code. Review it, approve it, then switch back to implementation.
02/compactAfter 30–45 min, context bloats. Compress the whole history into a focused summary of key decisions and current state so Claude stays sharp.
03/clearStarting a new task? Wipe the slate. Carrying a DB refactor's context into a frontend redesign produces conflicting code. One conversation per feature.
04/initScans your codebase and generates a CLAUDE.md — a persistent context doc (structure, tech stack, patterns, key decisions) read automatically every session.
05/costShows token usage for the session. Check it hourly during long sessions and budget against it — surprises are never fun.
06/memoryAdd persistent rules that apply in every session: "Always use TypeScript strict mode," "Always run tests after modifying /src/core."
07! prefixPrefix a message with ! to run it as a terminal command instead of sending it to Claude — quick tests, git status, navigation, without leaving the interface.
08Multi-Model SwitchingOpus for planning & architecture (thinks deeper, costs more); Sonnet for implementation (faster, cheaper). Plan with the thinker, build with the builder.

Productivity Techniques (09–18)

9The Reference File Technique

Don't describe the style you want — point at code that already has it.

prompt
Look at how authentication is implemented in src/auth/login.ts.
Implement password reset following the exact same patterns.

10The Screenshot Debug

UI looks wrong? Don't write a paragraph — screenshot it, paste with Ctrl+V, and point.

prompt
The button is misaligned with the input field. The spacing between
cards is inconsistent. Fix both.

11The Test-First Workflow

Tests define behavior before code exists — the implementation is correct because it must pass them.

prompt
Write tests for a function that calculates discounted prices. Cover:
normal discounts, zero discount, 100% discount, negative prices, and
string inputs. Then implement the function to pass all tests.

12The Incremental Build

Never say "build the entire feature." Break it into steps, testing between each: schema → test, API endpoint → test, validation → test, frontend form → test. Five small tested steps beat one massive prompt dramatically.

13The Codebase Question

Understanding before building prevents architectural mistakes.

prompt
Read src/services/ and explain how data flows from the API routes to
the database. What patterns are used? What should I know before
modifying anything here?

14The Diff Review

Catches the files Claude "helpfully" changed without being asked.

prompt
Show me a diff of every file you modified. Explain each change in one
sentence.

15The Error Paste

Copy the complete error and stack trace — not a summary. The step-by-step constraint stops Claude jumping to a wrong fix.

prompt
I got this error: [paste full error]. Diagnose the root cause step by
step before suggesting a fix.

16The Undo Checkpoint

Before every major change, commit. Reverting takes seconds instead of thirty minutes of debugging what used to work.

bash
git add . && git commit -m "checkpoint before [change]"

17The Parallel Session

For large features, open two terminals: one Claude Code session on the backend, one on the frontend. Each gets clean, focused context for its domain. Connect the pieces at the end.

18The Documentation Pass

Docs generated right after building are more accurate than docs written days later from memory.

prompt
Read every file you created or modified for this feature. Generate
comprehensive documentation: what each function does, how they connect,
expected inputs and outputs, and any non-obvious design decisions.

Architecture Techniques (19–26)

19The Architecture Audit

prompt
Analyze my project requirements: [list them]. Propose 2 different
architectural approaches. For each: component diagram, pros, cons,
estimated complexity, and what could go wrong. Recommend one with
clear reasoning.

20The Dependency Check

prompt
I want to add [package] to handle [use case]. Check: is this actively
maintained? Known security issues? Bundle size impact? Are there
lighter alternatives that cover my specific use case?

21The Pattern Enforcer

Add to your CLAUDE.md so Claude matches your patterns automatically in every new file:

CLAUDE.md
When creating new files, follow these patterns:
- API routes follow the structure in src/api/example-route.ts
- Database queries use the repository pattern in src/repositories/example-repo.ts
- React components follow src/components/ExampleComponent.tsx

22The Migration Builder

Schema changes that touch multiple layers are where Claude shines — it tracks all the ripple effects.

prompt
Change the user table schema: add a 'role' column (enum: admin, editor,
viewer; default viewer) and rename 'name' to 'display_name'. Generate
the migration, update the repository layer, update all API routes that
reference the old schema, and update the TypeScript types. Show me every
file that needs to change before making any modifications.

23The API Design Review

prompt
Review my API design: [paste route definitions]. Check for:
inconsistent naming, missing error responses, endpoints that should be
paginated, missing auth on protected routes, and REST convention
violations. Suggest specific improvements.

24The Security Scan

prompt
Scan this codebase for security vulnerabilities: SQL injection, XSS,
exposed secrets, missing input validation, insecure direct object
references, and missing rate limiting. For each finding: severity,
exact location, why it's dangerous, and the fix.

25The Performance Profiler

prompt
Analyze this codebase for performance issues: N+1 queries, missing
indexes based on query patterns, unnecessary React re-renders, large
bundle imports that could be lazy loaded, and endpoints that should be
cached. Prioritize by estimated impact.

26The Refactoring Planner

prompt
Read src/services/user-service.ts. It's grown to 800 lines and handles
too many responsibilities. Propose a refactoring plan that splits it
into focused modules. Show the proposed file structure, what moves
where, and verify no external imports break. Do NOT start refactoring
yet — just show me the plan.

Workflow Automation (27–31)

27The Git Hook Writer

prompt
Create a pre-commit hook that: runs the linter on staged files, runs
type checking, checks for console.log in production code, and blocks
commits that fail any check. Install it in .husky/pre-commit.

28The CI Pipeline Builder

prompt
Create a GitHub Actions workflow that: runs on every PR, installs
dependencies, runs the full test suite, runs the linter, builds the
project, and posts a comment on the PR with the results. Use caching
for node_modules.

29The Environment Setup Script

prompt
Create a setup.sh script a new developer runs once to set up the whole
dev environment: install dependencies, create .env from .env.example,
set up the local database, run migrations, seed test data, and verify
everything works by running the test suite.

30The Release Notes Generator

prompt
Read the git log since the last tag. Generate release notes organized
by: new features, bug fixes, performance improvements, and breaking
changes. Write each entry in user-friendly language, not developer
jargon. Format as a markdown changelog entry.

31The Database Seed Builder

prompt
Create a comprehensive seed file for the dev database: 5 users (1 admin,
2 editors, 2 viewers), 20 sample projects with realistic data,
relationships between entities, and edge cases (archived project,
deleted user, project with no members). Make the data realistic, not
'test123'.

Debug & Recovery (32–35)

32The Reproduction Prompt

prompt
This bug was reported by a user: [paste bug report]. Create a minimal
reproduction: exact steps, expected behavior, actual behavior. Then
write a failing test that captures this bug. Then fix the code to make
the test pass.

33The Blame Investigator

prompt
This function started failing yesterday. Read the git log for this file
over the past week. Identify which commit likely introduced the issue
and explain what changed. Then suggest the fix.

34The Dependency Conflict Resolver

prompt
I'm getting this dependency conflict: [paste error]. Analyze it.
Identify which packages require conflicting versions of the shared
dependency. Suggest the resolution that requires the fewest changes and
explain the tradeoffs.

35The Recovery Mode

When you've been going back and forth too long on a broken implementation:

prompt
Stop. Read the original working version of this file from git:
[paste git show output]. Now look at what we've been trying to achieve:
[restate the goal simply]. Start fresh with a different approach. The
previous approach clearly isn't working.
💡 Insight

Sometimes starting over from a clean state is faster than fixing accumulated mistakes. Know when to use this.

The Setup That Ties It All Together

When you start a new project, run this sequence — a five-minute setup that transforms every subsequent hour of development:

1/init — generate the CLAUDE.md file
2Add your coding standards and patterns to CLAUDE.md
3/memory — add persistent rules you want in every session
4Plan mode — design the architecture before writing any code
5Build incrementally — one feature at a time, tested at each step
