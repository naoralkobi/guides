How to Create Loops with Claude

AI · Agents · A nine-step build

The Big Idea

Stop making prompts. Start designing loops.

A prompt gets you one response. A loop gets you a system that keeps working after you close the laptop. Boris Cherny, who runs Claude Code at Anthropic, put it plainly: he doesn't prompt Claude anymore — he has loops running that prompt Claude and figure out what to do. His job is to write loops. Peter Steinberger framed the same shift differently: you shouldn't be prompting coding agents anymore, you should be designing loops that prompt your agents.

💡 Key insight

A loop is a recursive goal. You define a purpose, the agent iterates against it, and the loop runs until a real stopping condition is met. The agent forgets everything between runs. The loop does not. That single fact is the entire architecture.

What a Loop Actually Is

Addy Osmani, the Google engineer who wrote the essay that named this practice, breaks a loop into six parts. Every working loop is some combination of these.

PartWhat it does
AutomationsWhat makes a loop a loop instead of a one-time run — a schedule, cron job, webhook, or hook that fires without you typing anything.
WorktreesIsolated copies of the repo so parallel agents don't step on each other's files.
SkillsProcedure manuals the agent reads instead of being told from scratch every time.
ConnectorsLinks to the outside world the loop reads from and writes to.
Sub-agentsSpecialized agents that explore, implement, or verify on the loop's behalf.
MemoryA state file on disk — usually markdown — that survives between runs. The agent forgets; the file does not.

The Build

1Start with one trigger

Every loop starts with something that fires without you. The simplest version is a cron job that runs a Claude Code prompt on a schedule. The next version is a hook — a script that runs automatically when a specific event happens, like a commit or a file change.

Use case: pick one recurring task you currently do manually and turn the trigger into the first piece.

prompt 
Every morning at 8am, read yesterday's CI failures, open issues, and
recent commits, and write your findings to a markdown file.
💡 Insight

Don't try to build the full six-part system on day one. One automation that writes one state file is already more leverage than a hundred well-crafted prompts — because it runs without you.

2Give the loop a memory file

Create one markdown file — call it STATE.md or PROGRESS.md — and place it where every iteration can read and write it. This file is the loop's only memory. Everything the agent needs to pick up where it left off goes here.

At the start of each run the agent reads this file first; at the end it writes back what happened and what comes next. This is the PROGRESS.md pattern, and it's the single most important file in any loop. Without it, every run starts from zero regardless of how many runs came before.

PROGRESS.md 
## Done last run
- ...

## In progress
- ...

## Blocked
- ...

## Try next
- ...
⚠️ Watch out

Keep it short. A memory file the agent has to read 2000 lines of is worse than no memory file at all.

3Split the writer from the checker

The model that wrote the code is, in Osmani's words, too nice grading its own homework. A single agent that writes and then reviews its own work marks itself done more often than it should.

The fix is the evaluator-optimizer pattern, named in Anthropic's own writeup on building effective agents: one agent generates, a second critiques against an objective standard, and the loop repeats until the check passes. The check has to fail on something real — a test suite, a type checker, a build command, a linter.

⚠️ Watch out

A second agent told to "review this" with no objective signal just adds a second optimist. It will agree with the first agent more often than not. The verifier needs a hard gate, not an opinion.

4Isolate parallel work with worktrees

Once you're running more than one agent against the same codebase, isolation stops being optional. Give each agent its own working directory pointed at its own branch:

bash 
git worktree add ../agent-1-branch

A typical parallel setup: one sub-agent explores and writes a plan, a second implements against that plan in its own worktree, a third verifies the implementation against tests in a separate worktree. Each agent only ever sees its own copy. This is where loops scale from "one task running in the background" to "an entire pipeline running at once" — each isolated, each reporting back to the shared memory file when done.

5Set a hard stop condition

A loop without a real exit condition fails quietly. Engineer Geoffrey Huntley documented this as the "Ralph Wiggum loop": an agent meant to emit a completion signal only when finished emits it early, and the loop exits believing a half-done job is complete.

Your stop condition needs to be checkable by something other than the agent's own claim.

Real stop conditionNot a stop condition
The test suite passesThe agent says it's finished
The build succeeds 
The linked ticket moves to Done with a passing CI run 

Set a maximum iteration count as a backstop regardless of your primary condition — ten or twenty is reasonable for most loops. If the loop hits the ceiling without meeting the real stop condition, it should halt and flag for review, not keep running.

6Wire in a human review checkpoint

Not every loop should run fully unattended from day one. Boris Cherny's framing uses an autonomy ladder with four levels:

LevelWhat the loop does
1Suggests only.
2Drafts changes for a human to apply.
3Applies low-risk changes but requires human approval before publish or merge.
4Applies and completes automatically, with audit logs.

Start every new loop at level one or two. Run it for a week, read its output, and correct what it gets wrong. Once it consistently produces work you'd approve without changes, move it to level three. Level four is earned, not assumed.

💡 Insight

Runs that find something go to a triage inbox or flagged list. Runs that find nothing archive themselves silently. You should never have to open a loop's output to confirm that nothing happened.

7Watch the token cost

A single bad iteration is a wasted prompt. A single bad loop running unattended overnight is a bill. Agentic loops can run for dozens or hundreds of iterations, and each iteration is a full model call with the accumulated conversation history attached.

Before you let any loop run unsupervised, run it manually for three to five iterations and check the token usage per iteration. Multiply by your maximum iteration count for a worst-case cost per run; multiply that by how often the automation fires for a worst-case daily cost.

⚠️ Watch out

Build a command allowlist for any loop that can execute shell commands — restrict it to what the task actually needs (npm, git, ls, cat). An agent with unrestricted shell access inside an unattended loop is the fastest way to turn a token-cost problem into a security problem.

8Build the second loop differently than the first

Your first loop should be small, single-purpose, and heavily supervised. Your second loop should connect to the first. This is where automations, skills, and memory start compounding instead of just running in parallel.

A daily triage loop writes findings to a shared state file. A second loop, also on a schedule, reads that file and picks the highest-priority item to act on. Neither needs the other to function, but together they form a pipeline that moves work from "discovered" to "in progress" without you touching either one.

This is also when skills start paying off. Once you've written a skill file for how your loop should triage CI failures, every future loop that touches CI failures reads that same skill instead of you re-explaining it. The loops don't just run independently — they share what they've learned.

9Let your job become loop design

Once a few loops are running, your daily work changes shape. You stop opening a chat window to ask a question and start opening a triage inbox to review what the loops found overnight. The to-do list stops being a static pile of tasks and becomes a set of agents, routines, and loops that keep converting ideas into drafts, fixes, and reviews.

This doesn't mean you stop deciding what matters. It means the deciding happens at the loop-design level instead of the per-task level. You write fewer prompts not because you're doing less, but because the loops write them for you — and your attention moves to the parts that actually need a human: the review checkpoint, the stop condition, and the next loop worth building.

A prompt ends when the response arrives. A loop keeps working after you close the laptop. The leverage has moved from crafting the perfect message to building the system that sends messages for you, reviews the results, and decides what happens next.
